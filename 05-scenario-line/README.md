# 场景 / 容量设计过线（旁线）

目标：大厂若问「一亿邮件 / 海量推送 / 事件风暴」类题，**不哑火**。  
不是系统设计师岗刷题；要：**限流、队列、批处理、幂等、背压、可观测、回滚** + **对比选型**。

口令：可与 `编码练一题` 穿插；或说「练一道场景题」。  
作答模板：澄清约束 → 方案对比 → 主路径 → 瓶颈与降级 → 验证指标。

## SCE-01 一亿封邮件怎么发

**题目：** 业务要「短时间发约一亿封邮件/通知」，你怎么设计？有哪些方案对比？

**参考答法要点：**
- 澄清：时限、是否允许延迟、重复是否可接受、失败重试、是否要回执、供应商 QPS 上限
- 不要：单机 for 循环直连 SMTP；不要无界狂起 goroutine
- 主路径：生产写入队列（分片）→ worker 池按供应商限流拉取 → 批量 API → 失败进重试队列（退避）→ 幂等键防重
- 对比：同步直发 vs 异步队列；推 vs 拉；单队列 vs 按租户/优先级分片
- 容量：估算 QPS = 1e8 / 窗口秒；worker 数 ≤ 下游限额；水平扩容 worker
- 可观测：积压深度、成功/失败率、延迟、供应商 429；熔断与降级（只发重要）
- 与容器云挂钩：worker Deployment + HPA（看积压）；限流用令牌/租约；事件风暴自动化同构（限流/冷却/去重）

**追问：** 下游只允许 1k QPS 时如何排期？重复投递如何处理？

**对比点：** 队列削峰 vs 直接打爆；至少一次 vs 恰好一次（通常至少一次+幂等）  
**题源标签：** 经典场景变形 · 国内业务/平台面常见口吻  
**挂钩：** S3 事件风暴同构思维

## SCE-02 事件风暴下的自动处置

**题目：** Warning 事件暴涨，自动重启/扩容逻辑如何避免打崩集群？（深化 GO-D-09）

**要点：** 去重、冷却、速率限制、熔断人工、幂等、可行动 Reason 白名单；List-Watch 退避。  
**对比点：** 自动化止血 vs 控制器调谐；扩容 vs 限流  
**挂钩：** S3

## SCE-03 海量 Pod/节点下的「别把 apiserver 打挂」

**题目：** 自研运维组件要 List 全集群对象，有什么风险？怎么改？

**要点：** 禁止无分页狂 List；用 informer/share cache；限 QPS/Burst；字段选择与命名空间收敛。  
**对比点：** 每次全量 List vs List-Watch 缓存  
**题源标签：** 云原生 Go / 平台面常见

## SCE-04 限流与背压选型

**题目：** 入口流量突增，限流做在哪一层？令牌桶/漏桶/队列背压怎么选（口述级）？

**要点：** 网关/sidecar/业务/worker 消费侧；保护下游优先；队列积压要有丢弃/降级策略。  
**对比点：** 拒新流量 vs 缓冲延迟；客户端限流 vs 服务端限流
