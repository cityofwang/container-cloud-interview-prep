# 作答写法

训练与合面统一用下列结构。先选对模板，再填内容；不编造未做过的项目。  
横切：**对比/选型思维**贯穿始终（见 `MINDSETS.md`）——机制段或边界段尽量带一组对比。

## 1. 四段式（概念 / 原理 / Go 口述）

适用于「这是什么 / 为什么 / 怎么用」类题（含 Go 摸底口述）。

1. **结论先行**：一句话直接答问（先给答案，再展开）。
2. **机制要点**：2～4 个关键点（原理、数据结构、控制面行为），忌堆术语；**能对比则对比**（和易混点/另一方案差在哪）。
3. **生产挂钩**：一句真实场景或可验证现象（事件、指标、代码路径）；无强挂钩则讲清边界即可。
4. **边界与坑**：常见误用、追问预判、如何验证对错；选型题补「为何不选 B」。

自检：去掉第 3、4 段后，第 1、2 段是否仍能独立得分；若被问「和 X 比呢」是否接得上。

> 题库策略（2026-07-24）：查缺补漏、**宁多勿漏**；场景题（一亿邮件类）走旁线 `05-scenario-line/`，与面次并行。

## 2. 排障路径（故障 / 现象题）

适用于 CrashLoop、OOM、NotReady、泄漏、超时等。

1. **现象与影响面**：谁在报错、影响范围、是否在扩大。
2. **分层排查**：由外到内（业务可见 → 工作负载/事件 → 节点/资源 → 网络/存储/依赖），每层说「看什么、排除什么」。
3. **根因候选**：给 1～2 个最可能原因 + 如何证实/证伪。
4. **止血 → 修复 → 复盘**：先止损，再根治，最后防再发（告警、发布、容量）。

口述时用「我先…再…若…则…」的时间线，避免上来就猜单一结论。

## 3. STAR（经历 / 故事卡）

对应 `00-profile/STORY-CARDS.md`：情境 → 目标/任务 → 动作 → 结果；面试收尾可加一句反思。

- **S** 情境：背景与约束（混部/cgroup/事件/CSI/监控等真实做过的）
- **T** 目标：你要解决什么（干扰可控、利用率、自动处置等）
- **A** 动作：你具体做了哪几步（可被追问的细节）
- **R** 结果：可观测结果；能量化更好；没有数字就说清前后对比
- **反思（可选）**：边界、踩坑、若重来怎么做

规则：只讲真实做过的事；数字与系统名可脱敏。

## 4. 编码作答（Go / Shell 过线）

目标是过线表达，不是算法竞赛。

1. **澄清**：输入输出、约束、失败语义（超时？取消？部分失败？）。
2. **思路口述**：数据结构/并发模型/退出条件；先讲再写。
3. **落码**：可读优先；错误与 `context` 取消要有着落。
4. **自测与边界**：空输入、超时、并发竞态；口头给复杂度即可。

合面现场写不完时：保证主路径可跑通 + 口头补边界，优于半截花活。

## 4.5 场景 / 容量设计（过线）

适用于「一亿邮件、海量推送、别打挂 apiserver」等。见 `05-scenario-line/`。

1. **澄清约束**：时限、QPS 上限、能否重复、失败语义。  
2. **方案对比**：至少两组（如直连 vs 队列；推 vs 拉）。  
3. **主路径**：队列/分片/worker 池/限流/幂等/重试退避。  
4. **验证与降级**：积压、成功率、熔断；和容器侧 HPA/事件风暴同构可加分。

## 5. 训练时怎么写（会 / 半会 / 不会）

开卷题库与摸底清单统一三档，写在对应 `CHECKLIST.md` / `RESULT.md`：

| 档 | 含义 | 训练动作 |
|----|------|----------|
| **会** | 闭卷能按模板说完，追问主线不倒 | 保持；合面抽查即可 |
| **半会** | 主线对，但缺例子、边界或易被追问打穿 | 补口述稿；优先入错题本弱项 |
| **不会** | 说不清或明显错误 | **必入** `review/wrong-book.md`；先补概念再回炉 |

统计：会 + 半会 ≥ 80% 且不会项有口述补稿，再约闭卷模拟。单题模拟分 ≤2 或硬伤，同样入库错题本。
