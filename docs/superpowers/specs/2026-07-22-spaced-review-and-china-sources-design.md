## 1. 目标

1. **错题按遗忘曲线复现**：薄弱点在后续训练/面次中多次出现，避免「讲过就忘」。
2. **题源贴合中国地区真实面试**：锚定国内大厂/云厂商/政企云面经与岗位口径，避免纯翻译国外八股或脱离国内场景的题。
3. **招聘市场雷达**：定期（及按需）扫描国内招聘渠道中与学员画像匹配的 JD，提取能力关键词，补充题库考察方向。

## 2. 错题本与复现策略（方案 C：混合）

### 2.1 错题本

路径：`review/wrong-book.md`（实现阶段创建）

每条记录至少包含：

| 字段 | 说明 |
|------|------|
| id | 如 `R1-Q06` / `GO-D-03` |
| 来源 | 开卷自评 / 模拟 / 编码 |
| 首次错因 | 概念错 / 表达糊 / 无例子 / 硬伤 |
| 首次日期 | YYYY-MM-DD |
| 复现计划 | 下次模拟必带；+3 天回炉；+7 天回炉 |
| 复现结果 | 日期 + 分；连续 ≥2 次 ≥4 分可归档为「已巩固」 |

入库规则：

- 自评「不会」、模拟 ≤2 分、或存在硬伤 → **必入**
- 自评「半会」或模拟 =3 → **建议入**（默认入，学员可标跳过）
- 仅表达略生但主线对（4 分）→ 不入，除非学员要求

### 2.2 复现节奏（混合）

1. **下次模拟必带**：自错题本抽 2～4 道（优先最近未复现、且未巩固的）
2. **间隔回炉**：若距首次错误或上次失败复现已满约 **3 天** / **7 天**，且期间没有合面/单科模拟，则在开卷训练中插入 **「错题回炉」小测**（10～20 分钟，闭卷口述 2～4 题）
3. **跨面次体现**：进入二/三/四面题包与合面时，从错题本提升变形追问（同考点换问法），不只原题照念
4. **巩固出库**：同一题连续 2 次复现得分 ≥4 且无硬伤 → 标「已巩固」，不再强制抽；可自愿复习

对话口令新增：

- `错题回炉` — 按 3/7 天规则立刻小测  
- `看错题本` — 列出未巩固条目  

### 2.3 与合面/过关的关系

- 合面出题：在原配比之外，**额外**预留 2～4 道错题复现位（可挤占同科题量，不无限加长）
- 某面过关时：若错题本仍有「本面核心考点」未巩固 ≥3 条，复盘中必须安排回炉，不阻塞标过关，但 **下一合面前必须完成一次回炉**

## 3. 国内真实题源规则

### 3.1 优先场景与公司口径

出题与追问优先对齐国内常见岗位表达：

- 容器云 / 云原生运维 / 混部与资源：互联网大厂基础架构、云厂商（ACK 类）、政企云交付
- Go：云原生/平台 Go 为主，业务后端 Go 为辅（字节/腾讯/阿里/美团等面经高频方向：GMP、channel、context、逃逸、pprof；以及 client-go/控制器直觉）

### 3.2 题感要求（必须）

每道新题尽量满足至少一条「国内感」：

- 出现国内常见对象：ACK/TKE/托管集群、混部、CSI 插件发布、事件驱动运维、监控告警值班
- 或国内面经高频 Go 考点：GMP 口述、channel 坑、map 并发、context 取消、泄漏排查
- 追问用国内面试官口吻：「线上怎么查」「怎么和业务解释」「变更怎么灰度」

### 3.3 明确少用/不用

- 纯译国外博客、与国内栈无关的工具名堆砌
- 脱离生产的纯背诵定义且无追问路径
- 超纲中间件源码（除非摸底显示明显偏强且学员要求）

### 3.4 题源标注（实现时）

题包中可加一行 `题源标签`：如 `国内高频` / `面经变形` / `结合学员经历`。不必每题写公司名，避免编造「某厂原题」。

## 5. 招聘市场雷达（方案 D）

### 5.1 触发方式

- **定期**：约每周一轮（可用 Cursor 定时/提醒或会话内约定；实现阶段写入 `review/job-market/README.md` 操作说明）
- **按需**：学员发送口令 `扫岗位` 立即加跑一轮
- 两者并存（D）

### 5.2 匹配画像（检索关键词）

优先检索国内 JD（公开网页/招聘页/可访问摘要），关键词组合示例：

- 容器云 / 云原生 / Kubernetes / ACK / 混部 / CSI / 平台运维
- Golang / Go / 云原生开发 / 基础架构 / 管控面 / controller（辅：后端开发 Go）
- 年限口径：约 3–7 年或「资深/高级」中与学员经历可对标者（不硬套 title）

排除：纯前端、纯 Java 业务、与 K8s/Go 均无关的岗。

### 5.3 产出物

每次扫描写入：

`review/job-market/YYYY-MM-DD-scan.md`

固定结构：

1. **来源与局限**（哪些渠道成功、是否登录墙、样本数）
2. **命中岗位摘要**（公司类型脱敏可写「大厂/云厂商/中厂」+ 岗位名 + 3～8 条硬技能）
3. **高频能力词**（出现次数或主观权重）
4. **题库缺口**（现有容器/Go 题包未覆盖或覆盖浅的点）
5. **补充建议**（拟新增题方向 3～7 条；不一次写完整答案，待学员确认后再落题）

口令：

- `扫岗位` — 立即扫描并写报告  
- `看岗位雷达` — 打开最近一次报告并讨论补题  

### 5.4 渠道与能力边界（必须写进每次报告）

- **优先**：公开招聘页、公司官网社招、可抓取的职位描述、近期中文面经中的 JD 片段
- **尽力**：Boss 直聘 / 脉脉网页版公开片段（常需登录或反爬）
- **做不到时**：明确写「本轮未登录 App 内页」，改用公开源，不假装已刷完全站
- **禁止**：要求学员提供账号密码；不存储学员在招聘 App 的登录态到仓库

### 5.5 如何影响题库

- 雷达报告中的「补充建议」经学员一句确认（如「按报告补题」）后，再写入对应 `01-round*` / `06-golang/*` 小题包或「待补题清单」
- 不自动大批量改题，避免题库漂移失控
- 与 §3 国内题源规则一致：补题必须带国内场景感

## 6. 启动顺序（本增强的实现）

1. 用户确认本 spec + Go 双轨 spec  
2. 实现计划中包含：`review/wrong-book.md` 模板、`review/job-market/` 骨架与首轮扫描说明、README/口令更新、摸底题包（国内题感）、结构检查更新  
3. 摸底与之后每次模拟结束时，维护错题本  
4. 首轮 `扫岗位` 可在摸底前后任一时刻执行；建议实现完成后立刻跑一轮打底
