## 1. 目标

1. 用**可计算、可复盘**的评估，反映学员真实能力，而不是主观「感觉会了」。
2. **进阶模块延后解锁**：谈薪、故事商品化、简历/JD 对齐、压力合面、Go 作品指引等，仅在正确率/能力档达到门槛后再纳入训练与题库，避免基础不牢时分心。
3. 学员确认策略：**D** — 先不实现进阶内容；达标后再加入。

## 2. 能力维度（评估雷达）

实现阶段维护：`00-profile/ASSESSMENT.md`（每次摸底/合面后更新）。

| 维度 ID | 名称 | 考察来源 | 权重（合面综合） |
|---------|------|----------|------------------|
| C1 | 容器/K8s 基础运维 | 一面题、合面容器题 | 20% |
| C2 | 生产排障与可观测 | 二面向题、监控/事件故事 | 15% |
| C3 | 混部/cgroup/CSI 深度 | 故事卡 + 深挖追问 | 15% |
| G1 | Go 语言基础 | Go 摸底、basics | 10% |
| G2 | Go 并发与排障直觉 | concurrency、合面 Go 题、小编码 | 15% |
| G3 | 云原生 Go（client-go 等） | cloudnative 节 | 10% |
| X1 | 编码过线 | `06-golang/05-coding` + 合面现场题 | 10% |
| X2 | 结构化表达 | 模拟中的表达分（单独记，不抢技术主线） | 5% |

说明：

- 当前阶段 **不单独权重「谈薪/市场」**；解锁进阶后再加维度 M1。
- 每维得分 1.0～5.0（与题分同尺）；缺失数据标 `n/a`，不硬凑。

## 3. 分数如何计算（合理、可执行）

### 3.1 单题与卷面

- 单题 1～5 分（沿用既有量尺）
- **正确率（卷面）** = 该次测评中得分 ≥4 的题数 / 总题数  
- **有效均分** = 所有计分题平均分  
- **硬伤**：概念性错误且不自知 → 该题计 1，并强制入错题本

### 3.2 维度分

某维度分 = 近 **2 次**相关测评（开卷补考或模拟）中该维题目的有效均分；不足 2 次则用已有次数并标注「样本不足」。

### 3.3 综合档（给「像不像能拿更高薪」的粗标签）

| 档 | 条件（同时满足） | 含义（对标国内） |
|----|------------------|------------------|
| L1 打地基 | 综合加权 &lt; 3.0 或正确率 &lt; 50% | 未到稳定一面 |
| L2 一面可面 | 加权 ≥ 3.0 且正确率 ≥ 60%，无高频硬伤 | 可投对口初筛 |
| L3 合面稳定 | 加权 ≥ 3.5，正确率 ≥ 70%，错题未巩固 ≤ 5，C3 或 G2 至少一维 ≥ 3.5 | 冲中高级技术轮 |
| L4 可冲高薪技术面 | 加权 ≥ 4.0，正确率 ≥ 80%，C3≥4.0 且 (G2≥3.5 或 G3≥3.5)，X1≥3.5 | 技术定价上移的主依据 |
| L5 谈薪准备就绪 | 达 L4 + 进阶模块完成（谈薪包/故事/简历对齐） | 再开谈薪与投递加速 |

加权综合 = Σ(维度分 × 权重) / Σ(有数据的权重)。

### 3.4 评估节奏

- Go 摸底结束后：出首版 `ASSESSMENT.md`（G* 为主，C* 可先用容器一面自评填）
- 每次合面或错题回炉后：更新相关维 + 当前档
- 口令：`能力评估` — 输出当前雷达与距下一档缺口

## 3.5 评估标准的市场校准（与招聘雷达联动）

评估体系**不是一次性写死**。每次（约每周或 `扫岗位`）产出 `review/job-market/YYYY-MM-DD-scan.md` 后，必须多做一步：

### 校准流程

1. 从雷达报告提取：**高频硬技能、薪资/职级暗示、门槛词**（如必须 Go、必须有 Operator、只要运维不要开发等）
2. 对照当前八维权重与 L1–L5 门槛，回答：
   - 是否有维度被市场反复要求但权重过低？
   - 是否有维度市场几乎不问但我们考太重？
   - L3/L4 门槛是否明显低于/高于本周样本岗位的真实门槛？
3. 写入该次雷达报告新增一节：**「评估校准建议」**（可为空：写「本周无需调整」）
4. **默认不自动改标准**；需学员确认口令如 `按雷达校准评估` 后，才更新 `00-profile/ASSESSMENT.md` 中的「现行权重/门槛版本」与变更说明

### 可调整的对象（允许改什么）

| 可调 | 示例 | 约束 |
|------|------|------|
| 维度权重 | G2 从 15%→20%，C1 略降 | 八维之和仍为 100%；单次单维变动建议 ≤5 个百分点 |
| 解锁/档位附带条件 | L4 增加「G3≥3.5」若市场普遍要 controller | 不降低正确率主门槛（L3≥70%、L4≥80%）除非连续 2 周样本显示国内面明显更松/更紧并经学员确认 |
| 新增临时观察维 | 如「多集群/Service Mesh」记为 O1，样本不足不进加权 | 观察 ≥3 次雷达仍高频 → 再提议升为正式维 |
| 进阶模块门槛微调 | 如市场普遍要作品集，E4 提前到 L3 且 G2≥3.0 | E5 谈薪默认仍锁 L4，下调需学员显式同意 |

### 不可因单周噪声改的

- 取消错题回炉 / 国内题源 / 合面配比主框架
- 因单个 JD 要求「刷题 500」就把 X1 改成算法岗标准
- 在未达 L3 时强行解锁 E5 谈薪（与策略 D 冲突）

### 版本记录

`ASSESSMENT.md` 顶部维护：

- `标准版本`：如 `2026-07-22-v1`
- `最近校准`：日期 + 雷达报告链接 + 一句话变更

口令补充：

- `按雷达校准评估` — 采纳最近一次报告中的校准建议并改版本  
- `能力评估` — 按**现行版本**输出雷达与档位  

## 4. 进阶模块解锁准则（写入训练铁律）

下列模块 **默认锁定**，达标前不写入正式题库闯关主线、不合进合面必考：

| 模块 | 内容 | 解锁门槛 |
|------|------|----------|
| E1 故事商品化模板 | STAR 量化、可报价故事 | 达到 **L3**，且 C3≥3.5 |
| E2 简历 ↔ JD 对齐 | 雷达后的简历词表 | 达到 **L3**，且至少完成 **2 次** `扫岗位` |
| E3 压力合面 | 限时/打断/否定话术 | 达到 **L3**，且合面技术深度均分 ≥ 3.5 连续 2 次 |
| E4 Go 可见作品指引 | 小 controller/事件工具规格 | 达到 **L3**，且 G2≥3.5 或完成 concurrency 节过关 |
| E5 薪酬与谈薪包 | 带宽、期望、谈判话术 | 达到 **L4**（正确率与深度都够再谈钱） |

铁律：

1. **未解锁不主动展开**进阶材料（学员强要预习可给一页「预告」，不计入过关）。
2. 解锁后才把对应练习/题型编进「后面几面」与合面题单。
3. 正确率回落（连续 2 次合面正确率 &lt; 60% 或新增强硬伤）→ 进阶模块 **暂停**，先回炉错题与基础维。
4. 岗位雷达、错题回炉、国内题源、Go/容器双轨 —— **不在此锁内**，从一开始就运行。

## 5. 对实现的要求

- `00-profile/ASSESSMENT.md`：维度表 + 当前档 + 距下一档缺口 + 已解锁模块列表
- `README` 进度板：增加「能力档」「进阶解锁」
- 口令：`能力评估`
- 首期实现计划仍以：摸底、错题本、雷达骨架、Go 目录 为主；**E1–E5 仅写锁定说明，不建题包**

## 6. 范围确认

已确认：

- 进阶模块策略：**D**（正确率/档位达标后再加入）
- 评估体系：八维雷达 + L1–L5 档 + 明确解锁门槛
- E5 谈薪锁定到 **L4**；故事/简历/压力/作品锁定到 **L3**
- 评估权重与附带门槛：随每周/`扫岗位` 雷达给出校准建议，学员确认后再改版本
